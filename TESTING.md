# SMK-BOT — План тестирования

Документ описывает что нужно подготовить до тестирования, как тестировать каждый спринт (Этапы 4–12 из `PLAN.md`), и какие сценарии прогонять перед сдачей.

---

## 0. Подготовка инфраструктуры (до старта тестов)

### 0.1 Telegram-аккаунты и токены

| Что | Где взять | Куда положить |
|---|---|---|
| `TELEGRAM_API_ID` / `TELEGRAM_API_HASH` | https://my.telegram.org → API development tools (логин **по номеру userbot-аккаунта**, не личного) | `.env` |
| `TELEGRAM_PHONE` | Номер гретого userbot-аккаунта | `.env` |
| `BOT_TOKEN` | @BotFather → `/newbot` → имя «SMK Lead Bot» | `.env` |
| `OWNER_IDS` | @userinfobot → твой `user_id` (НЕ userbot, а твой основной) | `.env` |

**Подводный камень:** API_ID/API_HASH должны быть зарегистрированы на тот же номер, что и userbot. Иначе FloodWait при первом логине.

### 0.2 Тестовые чаты (минимум 3)

- **Receiver-чат** «Лиды-тест» — обычная группа.
  - Участники: твой основной аккаунт, userbot-аккаунт, **бот** (как админ — чтобы мог писать).
- **Source-чат №1** «Test SRO chat» — публичная или приватная группа.
  - Участники: userbot, твой основной аккаунт.
  - Назначение: туда шлём тестовые сообщения с ключами.
- **Source-чат №2** «Test noise chat» — для проверки отсутствия ложных срабатываний.
  - Шлём мусорные сообщения без ключей — лидов быть не должно.

**Лайфхак:** в source-чате №1 закрепить список тест-сообщений (см. раздел 6) для быстрого регресса.

### 0.3 Локальная инфраструктура

- Docker Desktop запущен.
- `psql` или DBeaver для отладки БД: `docker exec -it smk_bot-postgres-1 psql -U smk`.
- Telegram Desktop для основного аккаунта + мобильный/web для userbot.

### 0.4 Первая логинизация userbot (разовая)

Делается **на хосте**, не в Docker (нет stdin):

```bash
python -m app.userbot.client --login
# Telegram пришлёт код → ввести
# Если включён 2FA → пароль
# Создаст data/session.session
```

Дальше Docker подхватывает session через volume `./data:/app/data`.

### 0.5 Стартовый словарь

Подготовить файл `data/keywords_seed.txt` — 30–50 слов на старт (не 300, чтобы было управляемо в первых тестах):


Импортируется через `/importwords` после реализации Этапа 9.

## 1. Спринт A — Telethon-listener (Этап 4)

**Цель:** userbot подключается, слушает чаты из БД, при матче кладёт `Lead` в очередь.

### Сценарии

1. **Логин.** `python -m app.userbot.client --login` → ввести SMS-код → создался `data/session.session`.
2. **Старт.** `docker compose up` → в логе `userbot connected as @username` и `bot polling started`.
3. **Добавить чат вручную.** В psql:
   ```sql
   INSERT INTO chats (tg_id, title, active, added_by, added_at)
   VALUES (-1001234567890, 'Test SRO chat', true, <твой_id>, now());
   ```
   Перезапустить бот (или дождаться следующего reload_caches).
4. **Сматч.** В source-чате написать «нужно СРО». В `logs/bot.log` строка:
   ```
   matched chat=-1001234567890 author=... words=[сро]
   ```
5. **Без матча.** Написать «привет, как дела?» — в логе **ничего** про матч.
6. **Очередь.** `SELECT count(*) FROM lead_log` — пока 0 (sender ещё не реализован).

### Что проверяем особо

- `event.chat_id` для супергрупп с префиксом `-100...` — храним **как есть, с минусом**. В БД `tg_id` — `BigInteger`.
- При неактивном чате (`active=false`) — матч не происходит.

### Узкие места

- **FloodWait** при первом резолве чата (`get_entity`) — должен быть в `FloodWaitGuard`, не падать.

## 2. Спринт B — aiogram-каркас и админка чатов (Этапы 5, 7)

**Цель:** `/start`, инлайн-меню, управление чатами через бота.

### Сценарии

1. **Access control.**
   - OWNER пишет `/start` → видит главное меню.
   - Случайный юзер пишет `/start` → молчание (бот не отвечает).
2. **Меню.**
   - `📥 Чаты` → пусто, кнопка «➕ Добавить чат».
   - `🔑 Ключи` → пусто, кнопка «➕».
3. **Добавление чата через FSM.**
   - Жмём «➕» → бот: «Пришли ссылку или username».
   - Отправляем `@test_sro_chat` (или `https://t.me/test_sro_chat`).
   - Бот: «✅ Добавлено: Test SRO chat».
   - В БД появилась запись, `active=true`.
   - В Telegram: userbot физически вступил (проверить через клиент).
4. **Список чатов с пагинацией.**
   - Добавить 15 чатов скриптом → `📥 Чаты` → видим первые 10 + кнопка «▶».
   - Жмём «▶» → видим оставшиеся 5.
5. **Удаление.**
   - Жмём «✕» рядом с чатом → confirm «Точно удалить?» → «Да» → чат исчез из списка, в БД `active=false`, userbot физически вышел.
6. **Slash-команды дублируют UI.**
   - `/addchat @test_sro_chat` — работает.
   - `/listchats` — список.
   - `/removechat <id>` — удаление.

### Узкие места

- **FloodWait при join.** Если Telegram вернул FloodWait > 60 сек — бот отвечает «⏳ Telegram ограничил, попробую через N сек», задача уходит в фон.
- **Невалидная ссылка** — бот отвечает «❌ Не удалось добавить: <причина>», не падает.

## 3. Спринт C — Доставка лидов + слова (Этапы 6, 8)

**Цель:** полная цепочка: сообщение → лид в receiver-чате. Управление словарём через бота.

### Сценарии

1. **Установить receiver.**
   - `/setreceiver` → бот: «Пришли forward из целевого чата».
   - Форвардим любое сообщение из receiver-чата → бот: «✅ Получатель: Лиды-тест».
   - В `settings`: `receiver_chat_id` записан.
2. **Добавление ключа.**
   - `🔑 Ключи → ➕` → «сертификация ИСО» → выбрать категорию «Документация и ИСО» → выбрать substring → ✅.
3. **Полный сматч.**
   - В source-чате написать: «нужна сертификация ИСО 9001 срочно, есть кто?».
   - В receiver-чат пришёл лид по шаблону ТЗ-2 п.2.5:
     ```
     🔔 НОВЫЙ ЛИД | чат: «Test SRO chat»
     👤 Имя: <твоё имя>
     🔗 @username
     💬 Сообщение:
     «нужна сертификация ИСО 9001 срочно, есть кто?»
     🔑 Найденные ключевые слова: сертификация ИСО
     🕒 Время: 12.05.2026 14:17
     🔗 Ссылка: https://t.me/c/1234567890/123
     ```
4. **Тест exact-match.**
   - Добавить `НОК` через `➕`, выбрать **exact**.
   - Написать в source-чат «нужен НОК срочно» → лид пришёл.
   - Написать «кому нужно сделать» → лида **нет** (без exact «нок» нашлось бы как подстрока «кому»).
5. **Извлечение контактов.**
   - В source-чат: «Нужно СРО. Связь: +7 903 123-45-67, nikolay@mail.ru, @nik_test».
   - В лиде заполнены ВСЕ три поля: телефон, email, tg-username.
6. **Несколько ключей в одном сообщении.**
   - «Нужны СРО, НОК и ЭПБ» → в лиде `🔑 Найденные ключевые слова: сро, НОК, ЭПБ`.
7. **Длинное сообщение.**
   - Скинуть в source-чат текст ~5000 символов с ключом внутри → в лиде текст обрезан до ~4000 + «…», ссылка ведёт на полный.
8. **Antiflood.**
   - Написать 10 сообщений с ключами подряд → все 10 лидов приходят в receiver, без `RetryAfter`-ошибок в логе (между отправками задержка ≥ 1.1 сек).

### Узкие места

- **>4096 символов:** дефолт — обрезаем до 4000 + «…», полный текст по ссылке.
- **20 ключей сматчилось:** дефолт — все, разделитель `, `, при длине >200 символов — обрезаем «… и ещё N».
- **`TelegramRetryAfter`** — спим `retry_after + 1`, повторяем один раз.

## 4. Спринт D — Импорт/экспорт + админы + статус (Этапы 9, 10, 11)

**Цель:** заказчик может сам управлять словарём через .txt; OWNER может выдать админку коллеге; статус и пауза работают.

### 4.1 Импорт/экспорт

1. **Экспорт.** `/exportwords` → бот шлёт файл `keywords_2026-05-12.txt`.
   - Открыть в Notepad++ — UTF-8 **без BOM**.
   - Формат: одно слово на строке, `=` перед exact-словами, `| category` после.
2. **Импорт нового файла.**
   - Удалить в файле 3 строки, добавить 5 новых.
   - `/importwords` → послать файл.
   - Бот: «Найдено: новых 5, дубликатов 0, невалидных 0. Импортировать?».
   - «Да» → подтверждение → в БД +5 keywords, matcher перестроен.
3. **Roundtrip.**
   - `/exportwords` → ничего не меняем → `/importwords` тот же файл.
   - Бот: «новых 0, дубликатов всего».
   - Если хоть один «новый» — **баг** (несимметричный парсер).
4. **Невалидные строки.**
   - Файл с пустыми строками, `# комментариями`, дублями — все корректно отфильтрованы.

### 4.2 Управление админами

1. **Добавление.** OWNER:
   - `/addadmin` → «Пришли forward от целевого юзера или его user_id».
   - Forward'нуть сообщение от тестового аккаунта → бот: «✅ Админ добавлен: <full_name>».
   - В БД `admins` появился `role=admin`.
2. **Доступ нового админа.**
   - С тестового аккаунта `/start` → видит меню.
   - `📥 Чаты`, `🔑 Ключи`, `📤 Импорт` — работают.
   - **Кнопки «👮 Админы» НЕ видит** (только OWNER).
3. **Удаление.**
   - С OWNER: `/removeadmin <user_id>` → бот: «✅ Удалён».
   - С тестового аккаунта `/start` → молчание.
4. **OWNER неудаляем.**
   - В `/listadmins` рядом с OWNER'ами пометка `(owner)`, кнопки «✕» нет.
   - `/removeadmin <owner_id>` → бот: «❌ OWNER нельзя удалить через бота».

### 4.3 Статус и пауза

1. **`/status`** показывает:
   ```
   Статус: АКТИВЕН ✅
   Чатов: 12 (активных)
   Слов: 47
   Админов: 3 (OWNER: 1, ADMIN: 2)
   Получатель: «Лиды-тест»
   Очередь: 0
   Uptime: 2ч 14м
   Лидов за 24ч: 18
   ```
2. **`/stop`** → бот: «⏸ Мониторинг приостановлен».
   - В source-чат шлём «нужно СРО» → лида **нет**, в логе `monitoring disabled, skipping`.
3. **`/startbot`** → бот: «▶ Мониторинг возобновлён».
   - Следующее сообщение со словом → лид.
4. **Heartbeat.** Каждые 15 мин в `logs/bot.log` строка `heartbeat | chats=12 words=47 qsize=0`.

## 5. Спринт E — Стабилизация (Этап 12)

**Цель:** 48 часов uptime без падений, корректное восстановление после сбоев.

### 5.1 Reconnect

1. Отключить интернет на хосте на 2 минуты → включить.
2. В логе: `disconnected → reconnecting (attempt N, backoff Xs) → connected`.
3. Сообщения, пришедшие во время разрыва — Telethon делает `catch_up()`, лиды по ним приходят с задержкой.

### 5.2 FloodWait симуляция

1. Через бота добавить 10 чатов подряд (через `/addchat`).
2. Должен сработать `JOIN_CHAT_DELAY_SEC=25` (задержка между join'ами) или FloodWait.
3. Бот не виснет, очередь join'ов работает.
4. В логе: `flood wait Xs on JoinChannelRequest, sleeping`.

### 5.3 Перезапуск контейнера бота

1. `docker compose restart bot`.
2. Через 5 сек: лиды снова идут.
3. Все настройки сохранены: receiver, monitoring_enabled, список чатов и слов.
4. Матчер перестроен на старте (видно в логе `matcher rebuilt: N substring + M exact keywords`).

### 5.4 Перезапуск БД

1. `docker compose restart postgres`.
2. Userbot переживает (matcher в памяти).
3. aiogram переподключается (SQLAlchemy `pool_pre_ping=True`).
4. Запись новых лидов в `lead_log` возобновляется.

### 5.5 Бэкап

1. Запустить `scripts/backup.sh` → `backup_2026-05-12_14-00.tar.gz`.
2. В архиве: `pg_dump.sql` + `session.session` + `keywords_export.txt`.
3. Восстановление: распаковать на чистом хосте, запустить — система работает.

### 5.6 48-часовой прогон

1. Реальный uptime под нагрузкой 5–20 чатов.
2. По heartbeat'ам — `grep heartbeat logs/bot.log | wc -l` ≥ 190 (~48ч × 4/час).
3. Никаких `ERROR` или `CRITICAL` в логе кроме известных и обработанных.
4. `qsize` в heartbeat'ах не растёт монотонно (значит sender справляется).
5. RSS памяти процесса стабилен (нет утечек).

- [ ] Ложных срабатываний ≤ 15% от потока (мерим: число лидов, помеченных менеджером как «не релевантно», к общему за сутки)
